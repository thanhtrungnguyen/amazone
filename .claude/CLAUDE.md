# CLAUDE.md

## Project Overview

Full-featured e-commerce platform (Amazon-like) built as a **package-based Nx monorepo**.

- **Monorepo**: Nx (package-based — each project has its own `package.json`)
- **App**: Next.js 15 (App Router, full-stack with Server Components + Server Actions)
- **Database**: PostgreSQL + Drizzle ORM
- **Auth**: NextAuth.js v5
- **Payments**: Stripe
- **Styling**: Tailwind CSS + shadcn/ui
- **State**: Zustand (client), Server Actions (server mutations)
- **Validation**: Zod

## Architecture Rules

### Package Boundaries

- **Domain packages** (`products`, `cart`, `checkout`, `orders`, `users`, `reviews`) own their types, Zod schemas, server actions, and business logic
- **Domain packages import `@amazone/db`** for queries — they never import each other directly unless there's a real dependency (e.g., `checkout` depends on `cart` and `orders`)
- **`shared-ui`** contains components used by 2+ domains. Domain-specific components stay in the domain package or `apps/web/src/components/`
- **`shared-utils`** contains pure utility functions with no domain knowledge
- **`apps/web`** composes domain packages into pages — it imports from packages but packages never import from apps
- Use `@amazone/*` scope for all package imports

### Dependency Rules (Nx Enforced)

```text
apps/web → packages/* (any)
packages/checkout → packages/cart, packages/orders, packages/db
packages/cart → packages/products, packages/db
packages/orders → packages/products, packages/db
packages/products → packages/db
packages/reviews → packages/products, packages/db
packages/users → packages/db
packages/shared-ui → packages/shared-utils
packages/* → packages/shared-utils (any)
```

### Coding Standards

- **TypeScript strict mode** — no `any`, explicit return types on exported functions
- **Server components by default** — only add `"use client"` when interactivity is needed
- **Server Actions for mutations** — defined in domain packages, called from app pages
- **API routes only for**: Stripe webhooks, external integrations, REST access
- **Drizzle for all DB access** — never raw SQL outside Drizzle's `sql` tag
- **Zod schemas** co-located with domain types in each package
- **Money as integers** — store prices in cents, format to dollars only in UI
- **Use `integer` for prices** in Drizzle schema — never `real` or `doublePrecision`
- Each package has a barrel `index.ts` — consumers only import from the package root

### Naming Conventions

- Packages: `@amazone/<name>` (e.g., `@amazone/products`)
- Files: `kebab-case.ts` (e.g., `product-card.tsx`, `format-price.ts`)
- Components: `PascalCase` (e.g., `ProductCard`, `CartDrawer`)
- DB tables: `snake_case` plural (e.g., `products`, `order_items`)
- Server Actions: `camelCase` verbs (e.g., `addToCart`, `createOrder`)
- Zustand stores: `use<Name>Store` (e.g., `useCartStore`)

### E-Commerce Domain Rules

- Cart operations must be optimistic (update UI immediately, sync with server)
- Product prices always display with 2 decimal places
- Inventory checks happen at checkout, not just at add-to-cart
- Search supports fuzzy matching and category filtering
- All product images have alt text for accessibility
- Ratings display as stars with numeric average and review count

### Error Handling

- Use Next.js `error.tsx` boundaries for page-level errors
- Toast notifications (shadcn toast) for user action feedback
- Log server errors, return user-friendly messages to client
- Stripe errors: catch and map to user-friendly messages

### Performance

- Next.js Image component for all product images
- Cursor-based pagination for product lists
- React Suspense + `loading.tsx` skeletons for async content
- ISR for product listing pages (revalidate on product update)
- Prefetch product detail pages on hover

## Key Commands

- `pnpm nx dev web` — Start Next.js dev server
- `pnpm nx build web` — Production build
- `pnpm nx run-many -t build` — Build all packages
- `pnpm nx run-many -t lint` — Lint all packages
- `pnpm nx run-many -t test` — Test all packages
- `pnpm nx affected -t build` — Build only changed packages
- `pnpm nx affected -t test` — Test only changed packages
- `pnpm nx graph` — Visualize dependency graph
- `pnpm nx reset` — Clear Nx cache
- `pnpm drizzle-kit generate --config packages/db/drizzle.config.ts` — Generate migration
- `pnpm drizzle-kit migrate --config packages/db/drizzle.config.ts` — Run migrations
- `pnpm drizzle-kit studio --config packages/db/drizzle.config.ts` — DB GUI

## Package Manager

**Always use pnpm.** Never use npm or yarn.

- Install deps: `pnpm add <pkg>` (root) or `pnpm --filter @amazone/<pkg> add <dep>`
- Install all: `pnpm install`
- Run script: `pnpm run <script>` or `pnpm nx <target> <project>`
- Workspace file: `pnpm-workspace.yaml`

## Development Workflow (Plugins)

### Building Features — use `/feature-dev`

Use the **feature-dev** plugin for new e-commerce features. It analyzes the monorepo structure before generating code, ensuring proper package boundaries.

### Frontend Design — use `/frontend-design`

Use for all UI work: product cards, checkout flows, dashboards, landing pages. Produces polished Amazon-like designs.

### Code Review — use `/review-pr` or `/code-review`

Before merging any PR, run **pr-review-toolkit** for comprehensive review covering security, type design, silent failure detection, and test coverage.

### Git Workflow — use `/commit` and `/commit-push-pr`

- `/commit` — Atomic commits with conventional format
- `/commit-push-pr` — Commit, push, and open a PR in one step
- `/clean_gone` — Clean up stale local branches

### Security — security-guidance plugin (automatic)

Active on all code. Key areas: Stripe webhook verification, auth flows, user-generated content sanitization.

### Hookify — use `/hookify`

Configure rules to prevent: `.env` commits, `console.log` in production, `any` types, cross-package boundary violations.

### Simplify — use `/simplify`

After completing a feature, review for reuse opportunities and code quality.

## Multi-Agent Collaboration

This project includes specialized agents in `.claude/agents/` for team collaboration. Each agent has deep context about its domain, so any collaborator gets expert-level assistance.

### Available Agents

| Agent            | Domain                                          | Color  |
| ---------------- | ----------------------------------------------- | ------ |
| `db-architect`   | Schema design, Drizzle queries, migrations      | Blue   |
| `frontend-dev`   | Pages, components, layouts, shadcn/ui, Tailwind | Green  |
| `payments-agent` | Stripe, checkout, orders, webhooks              | Purple |
| `domain-logic`   | Business logic, server actions, Zod schemas     | Orange |
| `devops`         | Nx config, CI/CD, Docker, deployment            | Red    |
| `qa-tester`      | Unit tests, integration tests, Vitest           | Yellow |

### Collaboration Workflow

When multiple people work on the project with Claude Code:

1. **Clone the repo** — the `.claude/` folder comes with it, giving everyone the same agents, commands, and settings
2. **Use agents for domain work** — they enforce package boundaries and coding standards automatically
3. **Use commands for scaffolding** — `/setup`, `/new-page`, `/new-lib`, `/new-feature`, `/db-table` etc.
4. **Use plugins for quality** — `/commit` for git, `/review-pr` before merging, `/simplify` after coding

### Branching Strategy

- `main` — Production-ready, protected
- `develop` — Integration branch
- `feature/<name>` — Feature branches (e.g., `feature/product-search`)
- `fix/<name>` — Bug fixes
- `hotfix/<name>` — Production hotfixes

Use `/commit-push-pr` to create PRs from feature branches to `develop`.
