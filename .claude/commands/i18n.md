Set up or extend internationalization (i18n) for the amazone e-commerce platform.

Supported locales: English (en) and Vietnamese (vi).

Action (optional): $ARGUMENTS

You are an i18n expert — build a scalable translation system for an e-commerce platform.

## Architecture

### Routing Strategy
Use Next.js App Router i18n with middleware-based locale detection:

```text
apps/web/src/
├── app/
│   └── [locale]/              # Dynamic locale segment
│       ├── (shop)/            # Storefront pages
│       ├── (auth)/            # Auth pages
│       ├── dashboard/         # Seller dashboard
│       ├── layout.tsx         # Locale-aware root layout
│       └── page.tsx           # Home page
├── i18n/
│   ├── config.ts              # Locale config (en, vi), default locale
│   ├── dictionaries/
│   │   ├── en.json            # English translations
│   │   └── vi.json            # Vietnamese translations
│   ├── get-dictionary.ts      # Server-side dictionary loader
│   └── types.ts               # Dictionary type definitions
└── middleware.ts              # Locale detection + redirect
```

## Implementation Steps

### 1. Install Dependencies
```bash
pnpm --filter @amazone/web add next-intl
```
Or use the built-in Next.js i18n if preferred (no extra dependency).

### 2. Create Locale Config
In `apps/web/src/i18n/config.ts`:
- Define supported locales: `['en', 'vi']`
- Set default locale: `'en'`
- Export locale type for type-safe translations

### 3. Create Translation Files
- `apps/web/src/i18n/dictionaries/en.json` — English (base)
- `apps/web/src/i18n/dictionaries/vi.json` — Vietnamese

E-commerce categories to translate:
- Common UI (buttons, navigation, footer)
- Product details (price, stock, description, specifications)
- Cart & checkout (items, subtotal, shipping, payment)
- Auth (sign in, sign up, forgot password)
- Dashboard (orders, products, analytics)
- Error messages and notifications
- SEO metadata (page titles, descriptions)

### 4. Create Middleware
Detect locale from:
1. URL path prefix (`/vi/products`)
2. Cookie preference (returning users)
3. `Accept-Language` header (first visit)
4. Fall back to default (`en`)

### 5. Update Layouts
- Pass locale to root layout via `params`
- Set `<html lang={locale}>` attribute
- Load dictionary server-side (no client bundle bloat)

### 6. Create Translation Hook
For client components that need translations:
```typescript
// Server component: use getDictionary(locale)
// Client component: pass translations as props from server parent
```

## E-Commerce Specific Rules

### Currency Formatting
- **English (en)**: `$12.99` (USD, dot decimal)
- **Vietnamese (vi)**: `299.000 ₫` (VND, dot thousands separator)
- Use `Intl.NumberFormat` with locale-aware formatting
- Store prices in cents (USD) — convert for VND display

### Date Formatting
- **English**: `March 2, 2026`
- **Vietnamese**: `02/03/2026` (DD/MM/YYYY)
- Use `Intl.DateTimeFormat` with locale

### Number Formatting
- **English**: `1,234.56`
- **Vietnamese**: `1.234,56`

### SEO
- Add `hreflang` tags for alternate language versions
- Translate page `<title>` and `<meta description>`
- Use locale-prefixed URLs: `/en/products`, `/vi/products`
- Set canonical URL per locale

### Content Rules
- Product names and descriptions may be stored in both languages in the database
- Consider a `product_translations` table for multi-language product content
- Category names should be translated
- User-generated content (reviews) stays in original language
- Admin dashboard can be English-only initially

## Type Safety
- Generate TypeScript types from the English dictionary (source of truth)
- Vietnamese dictionary must match the same structure
- Use `satisfies` or strict typing to catch missing translations at build time
